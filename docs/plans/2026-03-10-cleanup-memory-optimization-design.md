# 2026-03-10 项目清理与内存优化设计

## 目标

在不改变项目功能、对外行为和运行方式的前提下，对代码做一次保守优化，重点降低不必要的内存占用、删除明确未使用代码，并提升整体可维护性与整洁度。

## 约束

- 采用保守优化策略，只做低风险改动
- 仅删除能够明确确认未使用、且无副作用的代码
- 不修改业务规则、信号逻辑、Telegram 命令行为
- 不引入新依赖
- 不进行大规模重构
- 以静态审查为主验证，不依赖真实外部请求验证

## 备选方案

### 方案 1：保守清理 + 局部内存优化（采用）
- 删除明确未使用代码
- 去掉明显无意义的中间变量、缓存或重复构造
- 缩短大对象存活时间
- 保持现有模块边界与主流程不变

### 方案 2：保守清理 + 小范围结构整理
- 在方案 1 基础上，进一步合并少量重复逻辑
- 风险略高于方案 1

### 方案 3：仅删除无用代码
- 风险最低，但内存优化收益有限

## 范围

### 重点检查模块
- `main.py`
- `strategy_sig.py`
- `notifier.py`
- `exchange_utils.py`

### 非重点模块
除非与清理直接相关，否则不修改以下文件：
- `README.md`
- `Dockerfile`
- `docker-compose.yml`
- `.env.example`
- `CLAUDE.md`

## 优化内容

### 会做
- 删除未使用的函数、变量、导入
- 去除无意义中间变量
- 删除不会被消费的缓存结构
- 缩短大对象作用域
- 收敛小范围、完全等价的重复逻辑

### 不会做
- 不改业务逻辑
- 不改信号判定规则
- 不改 Telegram 交互语义
- 不做架构级重构

## 验证方式

- 基于静态审查确认删除内容未被调用
- 复查 import、调用链和数据流
- 对每项修改说明其安全性
- 必要时仅做非常有限的语法级验证

## 执行顺序

1. 盘点未使用代码与冗余对象
2. 先做确定性的删除
3. 再做局部内存优化
4. 输出变更摘要与风险说明
